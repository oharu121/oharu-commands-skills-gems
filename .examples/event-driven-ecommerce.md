# イベント駆動型 ECサイト注文システム

## 概要

ECサイトの注文処理をイベント駆動アーキテクチャで構築したサーバーレスシステムです。注文の受付からキュー処理・出荷・通知まで、各ステップが疎結合に連携し、負荷変動に強いスケーラブルな構成を実現しています。

## コンポーネント一覧

| AWSサービス | 図中のラベル | 役割 | 選定理由 |
|---|---|---|---|
| Amazon CloudFront | CloudFront | 静的コンテンツ配信・APIキャッシュ | グローバルな低レイテンシ配信、DDoS対策の第一層 |
| Amazon API Gateway | 注文受付 | REST APIエンドポイント | Lambda統合、リクエスト検証・スロットリングを標準装備 |
| AWS Lambda（注文処理） | 注文処理 | 注文リクエストの検証・SQSへの送信 | サーバーレスで瞬時にスケール、受付と出荷を分離して耐障害性を向上 |
| Amazon SQS | 注文キュー | 注文メッセージの一時保管・配信保証 | 注文の急増時にバッファとして機能、メッセージの損失を防止 |
| AWS Lambda（出荷処理） | 出荷処理 | SQSからメッセージを取得し出荷処理・DB保存 | SQSトリガーで自動実行、同時実行数で処理レートを制御可能 |
| Amazon DynamoDB | 注文DB | 注文データの永続化 | ミリ秒レベルの読み書き性能、DynamoDB Streamsでイベント駆動の起点に |
| Amazon EventBridge | イベント検知 | DynamoDB Streamsからのイベントルーティング | 複数の通知先やダウンストリーム処理への柔軟なルーティングルール |
| Amazon SNS | 通知サービス | お客様への注文確認・出荷通知の送信 | メール・SMS・モバイルプッシュなど複数チャネルへの同時配信 |

## データフロー

### 注文処理フロー（①〜⑥）

お客様が商品を注文してから、出荷処理・注文データの保存までの流れです。

| ステップ | 説明 |
|---|---|
| ① 注文 | お客様がECサイトで注文ボタンをクリック。CloudFrontが最寄りのエッジでリクエストを受信 |
| ② 受付 | API Gatewayがリクエストを検証（必須パラメータ・フォーマットチェック）し受付 |
| ③ 処理 | Lambda（注文処理）が注文内容を検証（在庫確認・金額計算）し、SQSにメッセージとして送信 |
| ④ キュー | SQSが注文メッセージを安全に保管。注文が殺到しても、キューがバッファとなり後続処理を保護 |
| ⑤ 出荷 | SQSがLambda（出荷処理）をトリガー。出荷指示の作成・在庫の引き当てを実行 |
| ⑥ 保存 | 処理結果をDynamoDBに永続化。注文ステータス「出荷準備中」として記録 |

### 通知フロー（❶〜❸）

注文状態が変更されたとき、お客様にリアルタイムで通知する流れです。

| ステップ | 説明 |
|---|---|
| ❶ 変更検知 | DynamoDB Streamsが注文レコードの変更を検知し、EventBridgeにイベントを送信 |
| ❷ 通知送信 | EventBridgeがイベント内容に基づきルーティング。SNSへ通知リクエストを転送 |
| ❸ お届け | SNSがお客様のメール・SMS・モバイルプッシュに注文確認や出荷通知を配信 |

### フロー全体の特徴

注文処理フロー（①→⑥、左→右）と通知フロー（❶→❸、右→左）が**ループ構造**を形成しています。お客様から始まった注文がシステムを通過し、通知として再びお客様に届くまでの一連のサイクルを表現しています。

## 設計上のポイント

- **SQSによる疎結合**: 注文受付（Lambda①）と出荷処理（Lambda②）をSQSで分離。受付側は即座にレスポンスを返し、出荷処理は自分のペースで処理できる。一方が障害を起こしても他方に影響しない
- **イベント駆動の通知**: DynamoDB Streams → EventBridge → SNS のチェーンにより、注文状態の変更が自動的にお客様への通知にトリガーされる。新しい通知先の追加はEventBridgeのルール追加だけで完了
- **完全サーバーレス**: Lambda + SQS + DynamoDB + EventBridge + SNS のすべてがフルマネージド。サーバーの運用・パッチ適用が不要
- **スケーラビリティ**: セール時の注文急増もSQSがバッファとして吸収。Lambda の同時実行数で出荷処理のレートを制御可能

## コスト・スケーリングに関する補足

- **Lambda**: リクエスト単位の従量課金。注文がない深夜帯はコストゼロ
- **SQS**: メッセージ送受信の従量課金。最初の100万リクエスト/月は無料枠
- **DynamoDB**: オンデマンドモードなら読み書き単位の従量課金。プロビジョンドモードならキャパシティ予約で割安に
- **EventBridge**: イベント単位の従量課金（$1.00/100万イベント）。低コスト
- **SNS**: 通知チャネルにより異なる。メールは最初の1,000通/月無料、SMSは送信先の国により従量課金
